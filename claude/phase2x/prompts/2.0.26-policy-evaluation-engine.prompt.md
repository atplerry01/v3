SECTION 2 — COMPONENT PURPOSE
------------------------------------------------------------

The Policy Evaluation Engine evaluates policy definitions against a given PolicyContext.

This engine represents the core **WHYCEPOLICY™ governance decision engine**.

Responsibilities:

• evaluate policy conditions
• determine policy actions
• enforce policy priority rules
• produce deterministic policy decisions

The engine must support evaluation of multiple policies in deterministic order.

Policy ordering is determined by the Policy Dependency Engine.

SECTION 7 — EVALUATION ALGORITHM
------------------------------------------------------------

Create file:

PolicyEvaluationEngine.cs

Primary method:

EvaluatePolicies(PolicyEvaluationInput input)

Algorithm:

1 iterate policies in deterministic order
2 skip policies with lifecycle state != ACTIVE
3 evaluate policy conditions against context attributes
4 if conditions satisfied evaluate policy action
5 append PolicyDecision record
6 apply priority override logic
7 determine final decision

Rules:

• deterministic evaluation
• stable ordering
• priority override supported

------------------------------------------------------------
SECTION 8 — PRIORITY RULES
------------------------------------------------------------

PolicyPriority values:

LOW
MEDIUM
HIGH
CRITICAL

Higher priority policies override lower priority ones.

Example:

DENY at HIGH priority overrides ALLOW at LOW priority.

------------------------------------------------------------
SECTION 9 — ENGINE RULES
------------------------------------------------------------

This engine must follow WBSM v3 execution rules.

The engine must be:

• stateless
• deterministic
• thread-safe

The engine must NOT:

• persist data
• call other engines
• access infrastructure
• mutate domain aggregates

All evaluation must occur using the provided input models.

------------------------------------------------------------
SECTION 10 — EVENT BEHAVIOR
------------------------------------------------------------

This engine does not emit domain events.

Policy decisions are returned to the runtime pipeline.

Runtime workflows may then trigger events based on the decision.

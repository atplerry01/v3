# WHYCESPACE WBSM v3
# PHASE 2.0.29 — POLICY CONFLICT DETECTION ENGINE

You are implementing Phase 2.0.29 of the Whycespace system.

This phase introduces the **Policy Conflict Detection Engine**.

The engine analyzes policies and detects conflicts where two or more policies produce contradictory outcomes.

Conflict detection is required before policies can be safely deployed.

---

# ARCHITECTURE RULES

ENGINE → stateless logic  
SYSTEM → stateful storage

The conflict detection engine must be stateless.

It may read from policy stores but must not modify them.

# MODEL

Create file:

PolicyConflict.cs

Represents a detected conflict between policies.

Structure:

public sealed record PolicyConflict(
    string PolicyA,
    string PolicyB,
    string Domain,
    string Reason,
    DateTime DetectedAt
);

Reason examples:

Contradicting actions

Example:

PolicyA → allow
PolicyB → deny

---

Create file:

PolicyConflictReport.cs

Represents the output of conflict analysis.

Structure:

public sealed record PolicyConflictReport(
    string Domain,
    IReadOnlyList<PolicyConflict> Conflicts,
    DateTime GeneratedAt
);

---

# STORE

No store required.

Conflict detection is analytical and does not persist results.

---

# ENGINE

Create file:

PolicyConflictDetectionEngine.cs

Responsibilities:

Analyze policies and detect contradictions.

Conflict scenarios:

1 Contradicting actions

Example:

PolicyA → allow
PolicyB → deny

2 Duplicate conditions with different actions

Example:

WHEN trust_score < 50
THEN allow

WHEN trust_score < 50
THEN deny

3 Dependency conflict chains

Example:

PolicyA depends on PolicyB
PolicyB denies PolicyA conditions

---

# METHODS

DetectConflicts(string domain)

Steps:

1 Retrieve policies from PolicyRegistryStore

2 Filter by domain

3 Compare policies pairwise

4 Detect contradictory actions

5 Generate PolicyConflictReport

---

# ACTION CONFLICT RULES

Conflicts occur when:

allow vs deny

deny vs allow

Other actions (log, notify, flag) are not considered hard conflicts.

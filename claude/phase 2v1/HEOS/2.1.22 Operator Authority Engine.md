# 2.1.22 OPERATOR AUTHORITY ENGINE

You are implementing the **Operator Authority Engine** inside the Whycespace system.

The Operator Authority Engine determines whether an operator has sufficient authority to perform a specific operational action.

This engine enforces **operational governance rules** within HEOS.

Examples of governed actions:

• workforce assignment approval  
• workforce suspension  
• schedule override  
• operational policy enforcement  
• SPV supervision  

The engine evaluates the **OperatorAggregate** defined in:

src/domain/core/operators/

This implementation must follow **WBSM v3 Prompt Structural Rules and Guardrails**.

# STEP 7 — AUTHORITY RULES

The engine must enforce:

Operator must have **Active status**.

Operator must possess scope covering the target action.

OperatorLevel determines allowable actions.

Example hierarchy:

Supervisor → limited workforce actions  
Manager → workforce + scheduling  
Director → SPV-level authority  
SystemOperator → global authority

Suspended operators cannot perform actions.

Revoked operators always fail authorization.

---

# STEP 8 — ENGINE INVOCATION RULE

The engine must never call other engines.

Correct invocation flow:

Runtime Dispatcher → OperatorAuthorityEngine

Incorrect:

OperatorAuthorityEngine → another engine

Runtime orchestration must invoke engines.

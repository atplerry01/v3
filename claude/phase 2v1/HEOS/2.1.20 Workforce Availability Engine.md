# WHYCESPACE WBSM v3
# HEOS — HUMAN ECONOMIC OPERATING SYSTEM
# 2.1.20 WORKFORCE AVAILABILITY ENGINE

You are implementing the **Workforce Availability Engine**.

This engine manages **availability state for workforce members** in the HEOS system.

Availability determines whether a worker can be assigned to economic tasks.

Examples:

• Available  
• Busy  
• Offline  
• OnLeave  

This engine operates on the **WorkforceAggregate** defined in:

src/domain/core/workforce/

The implementation must follow **WBSM v3 Prompt Structural Rules and Guardrails**.

---

# STEP 1 — ARCHITECTURE VERIFICATION

Component classification:

Workforce Availability Engine = **Execution Engine**

Correct layer:

src/engines/T2E/

Reason:

Availability management is deterministic business logic applied to the WorkforceAggregate.

It does not orchestrate workflows and does not manage distributed execution.

Therefore it belongs in **T2E Execution Engines**.

# STEP 6 — ENGINE IMPLEMENTATION

Create:

WorkforceAvailabilityEngine.cs

• update workforce availability state  
• enforce availability rules  
• apply changes to WorkforceAggregate  
• return decision results

Example method:

UpdateAvailability(WorkforceAggregate workforce, WorkforceAvailabilityCommand command)

Return:

WorkforceAvailabilityDecision

Engine requirements:

stateless  
thread-safe  
deterministic  

The engine must NOT:

persist state  
call other engines  
invoke infrastructure

---

# STEP 7 — AVAILABILITY RULES

The engine must enforce the following rules:

Worker must have **Active workforce status**.

Suspended workers cannot change availability.

Worker cannot move directly from Offline to Busy.

OnLeave workers cannot become Available until leave ends.

Timestamp must be valid.

---

# STEP 8 — ENGINE INVOCATION RULE

The engine must never call other engines.

Correct invocation flow:

Runtime Dispatcher → WorkforceAvailabilityEngine

Incorrect:

WorkforceAvailabilityEngine → another engine

Runtime orchestration must invoke engines.
